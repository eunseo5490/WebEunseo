# 스키마를 사용하는 이유

1. 데이터 구조를 정리하기 위해
  - 구조를 정해서 일관성 유지

2. 데이터 관리의 단위로 사용하기 위해
  - 프로젝트 별로 DB를 나눌 수 있다.
  - 예 : 쇼핑몰 / 게시판 / 회원관리

3. 접근 권한 관리
  - 특정 스키마만 접근하도록 설정 가능

2. 개념 스키마

# Data, DB, DBMS의 차이점
Data : 가공되지 않은 단순한 사실이나 값
DB : 데이터를 체계적으로 저장해놓은 공간
DBMS : Data Base Management System, 데이터베이스를 관리하는 프로그램/소프트웨어


# DBMS의 종류

1. 계층형 : 트리 구조로 데이터를 저장하는 방식
  - 1대N 관계만 허용
  - 구조가 단순함
  - 변경이 어려움

2. 네트워크형 : 데이터를 그래프 형식으로 연결하는 방식
  - N대N 관계 가능
  - 계층형보다는 유연함
  - 구조가 복잡함
  - 여러 개의 연결을 허용

3. 관계형 : 테이블 형태로 데이터를 저장하는 방식
  - 가장 많이 사용됨
  - SQL을 사용
  - 데이터의 무결성을 보장함

4. 객체지향 : 객체(Object) 형태로 데이터를 저장
  - Java, C++의 객체 그대로를 저장 가능
  - 복잡한 데이터 처리에 유리
  - 사용 빈도 낮음

5. NoSQL(Not only SQL) : 관계형이 아닌 다양한 방식의 DB
  - key <-> value
  - Document : JSON
  - Column : 컬럼 중식 저장
  - Graph : 관계 중심
  - 대용량 처리에 강함
  - 속도 빠름
  - 유연한 구조


# SQL(Structured Query Language)
- DBMS 상에서 데이터를 읽고 쓰고 삭제하는 등 데이터를 관리하기 위한 일종의 프로그래밍 언어

DDL(Data Definition Language) : 데이터 정의어
  - 데이터베이스 객체를 관리하는 언어
  - 테이블, 속성, 제약조건, 뷰
  - CRATE : 데이터베이스 객체 생성
  - DROP : 삭제
  - ALTER : 수정
  - TRUNCATE : 테이블은 유지, 데이터는 통 째로 삭제

제약조건
- 문제가 되는 데이터를 테이블에 추가할 수 없도록 제한하는 조건
- 테이블을 만들면서 설정할 수 있고, 추후에 생성 또는 변경 가능

PRIMARY KEY(기본키)
- 데이터에 값이 있어야 하고, 겹쳐서는 안된다.

NOT NULL
- 해당 속성에는 반드시 값이 들어가야 한다.

UNIQUE
- 속성에 들어가는 값은 유일해야 한다.

FOREIGN KEY(외래키)
- 다른 테이블의 기본키 값을 참조하는 제약조건

CHECK
- 속성에 들어갈 수 있는 데이터의 조건을 설정하는 제약조건

DEFAULT
- 값을 입력하지 않았을 때 자동으로 들어가는 기본값을 설정

데이터베이스 객체의 수정
ALTER

삭제

변경

자료형 변경

ALTER TABLE 테이블명 ADD 제약조건 종류(컬럼);

ALTER TABLE 테이블명 DROP 제약조건 종류(컬럼);

무결성
- 데이터베이스에 저장된 데이터가 현실 세계의 실제값과 일치하는 정확성을 의미한다.

1. 개체 무결성
- PK로 설정된 컬럼은 반드시 고유한 값을 가져야 한다. 그리고 NULL이 될 수 없다.

2. 참조 무결성
- 외래키(FK) 값은 NULL이거나, 반드시 참조하는 테이블의 PK에 들어있는 값과 일치해야 한다.
- 존재하지 않는 값은 외래키로 저장할 수 없다.

3. 도메인 무결성
- 특정 속성의 값이 미리 정의된 유효한 값 범위 안에 있어야 한다.

4. 사용자 정의 무결성
- 사용자가 논리적으로 필요한 규칙을 직접 정의하는 것
- 나이는 음수가 들어올 수 업삳. 가입일은 오늘 이후일 수 없다.

-- AND : 두 가지 이상의 조건을 비교 (그리고)

from film
where rating = 'PG' or rating = 'G'
