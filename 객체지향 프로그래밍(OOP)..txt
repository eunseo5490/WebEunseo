객체지향 프로그래밍(OOP).
객체지향이 아닐 때의 문제점
- 데이터와 기능이 따로 논다.
- 코드가 많아지게 되며 관리가 어렵다
- 재사용이 힘들다

객체지향의 핵심 개념
1. 클래스(class)
- 설계도
ex) 이 프로그램에서 사람을 어떻게 정의할 건지, 제품을 어떻게 정의할건지
클래스 안에 정의할 수 있는 내용
- 필드(field) : 각 개체가 가지는 정보(상태, 속성)
ㄴ 이름, 나이, 키, 몸무게, 혈액형, 성별, 주소, 학번 ...
- 메서트(method) : 각 객체가 할 수 있는 동작(기능)
ㄴ 공부한다, 먹는다, 운동한다, 잔다 ...

2. 객체(object)
클래스명 변수명 = new 클래스명();
- 클래스를 통해서 만들어진 것
- 필드에 객체만의 정보를 넣거나 필드의 내용을 사용할 수 있다.
객체변수명.필드명
- 클래스에 정의된 기능을 호출하여 사용할 수 있다.
객체변수명.메서드명();

3. 객체지향의 4대 특징
- 캡슐화
데이터를 보호하는 개념
필드 직접 접근하지 못하게 한다.
제공된 메서드를 통해서만 접근할 수 있게 한다.

- 상속
기존 클래스의 내용을 물려받는 기능(필드, 메서드)
코드를 재사용할 때 좋음

- 다형성
하나의 타입으로 여러 객체를 다룰 수 있는 성질
타이어마다 성능이 다르다.

- 추상화
클래스를 단순화 시키는 것

메서드 내부에서 메서드를 정의할 수 없다.
메서드 외부에서 메서드를 호출하는 것은 불가능하다.


참조용 클래스(객체를 만들기 위한 클래스)
- main 메서드 없음
- 필드
- 생성자 (호출했을 때 메모리에 객체를 올려주는 기능)
ㄴ 객체를 생성했을 때 필드를 초기화 하는 역할
정의하지 않으면 컴파일러가 기본생성자를 만들어서 사용

사용자가 직접 정의하면 컴파일러가 만든 기본생성자는 사용되지 않는다.

메서드처럼 매개변수를 가질 수 있다.

- 메서드 (여러 개 정의 가능)
접근제한자 반환타입 메서드명(매개변수){

}

메서드 (계산기)
정의 (기능을 만드는 것)
ㄴ 숫자 두 개와 연산자를 받을 준비
호출
ㄴ 숫자와 연산자를 전달하여 연산결과를 돌려받는다.


return
매개변수, 메서드 내부에서 정의된 변수는 지역변수이다.
메서드가 호출돼서 실행한다음 종료가 되면 메모리에서 삭제된다.

만약, 메서드 밖에서 연산결과를 사용하고 싶다면 return을 통해 값을 밖으로 꺼내야 한다.

메서드명
- 메서드를 호출할 때 사용하는 이름
- 첫글자는 소문자로 작성

//생성자(or메서드)를 호출한 객체의 필드임을 명시하기 위해서 this키워드를 사용한다.



오버로딩(overloading)
- 같은 이름으로 서로 다른 개수, 타입의 매개변수를 각각 처리하는 방법

- 반환형은 달라도 상관 없음
- 메서드(or생성자)명은 같아야 한다.
- 매개변수의 개수가 다르거나, 타입이 다르거나, 순서가 달라야 한다.

 

접근제한자(Access Modifier)
public : 프로젝트 내에서는 어디서든지 사용 가능
default : 같은 패키지 내의 다른 클래스 어디서든지 사용 가능
private : 현재 클래스에서만 사용 가능
protected : (상속 후)

필드/생성자/메서드/클래스에 붙힐 수 있다.



디자인 패턴
프로그래밍을 하다보면 비슷한 문제가 반복적으로 나타나는 경우가 있다.
- 객체를 하나만 만들어야 하는 경우
- 객체 생성이 너무 복잡한 경우
- 기능은 비슷한데 구현 방법이 여러개인 경우
- 객체들끼리 너무 강하게 연결되어 있어 수정이 어려운 경우

개발자들이 오랜 시간 동안 많이 쓰인 설계 방법을 정리해둔 것이 디자인 패턴이다.



상속의 특징
하나의 부모는 여러 클래스한테 상속할 수 있다.
하나의 자식이 여러 클래스를 상속받는 건 불가능


패키지
단순히 클래스를 분리하는 용도가 아니라 클래스별 구분자역할도 한다.
패키지 계층구조로 만들 수 있다.
com
  ㄴ kumho
    ㄴTire
  ㄴ hankook
    ㄴTire

보통은 회사 도메인의 역순으로 패키지 이름을 정한다.
도트(.)를 기준으로 계층이 생긴다.
com.kumho.Tire


코드를 작성할 때 완벽하게 작성
수정에 용이한 코드가 좋은 코드
너무 구체적인 코드는 좋지 않음
기능이 추가될 때 수정을 최소화하는 코드


싱글톤 패턴
객체를 메모리에 한 번만 올리고, 올라간 객체를 계속 사용하는 방식
1. 생성자를 private으로 정의해서 외부에서 호출하지 못하도록 한다.
2. 클래스 내부에서 static으로 객체를 하나 생성한다. (private)
4. 클래스 내부에서 만든 객체를 사용할 수 있도록 메서드(public)를 만든다.



상속
부모 클래스의 필드나 메서드를 자식 클래스에서 사용할 수 있는 것
자식 클래스가 부모 클래스를 고른다.



오버로딩(메서드의 중복 정의)
오버라이딩(메서드의 재정의)



클래스 타입변환
- 상속관계에서만 가능
자동타입변환 : 자식 -> 부모타입
- 자식클래스에 정의한 멤버를 사용할 수 없다.
강제타입변환 : 부모타입 -> 자식타입
- 자식클래스에 정의한 멤버를 사용할 수 있다.
(자식클래스)부모객체



생성자 : 객체를 메모리에 올리는 역할, 필드를 초기화하는 역할
메서드 : 만들고 싶은 기능, 자주 사용하는 기능을 만드는 것

this : 호출한 객체 자신
this.필드명
this()

super : 부모클래스
super.필드명
super()
super.메서드명()

오버라이딩(메서드 재정의)
상속관계에서만 사용 가능

부모클래스에 있는 메서드 이름으로 사용하고 싶지만
자식클래스의 상황에 맞지 않아서 자식클래스에서 재정의할 때

접근제한자, 반환형, 메서드명, 매개변수는 그대로 사용
내용만 수정

@Override -> 어노테이션
컴파일러한테 오버라이드된 메서드라고 알려주는 것


다형성 : 상속 + 오버라이딩 + 타입변환
부모타입의 변수에 자식객체들이 들어올 수 있다.
부모타입으로 자식객체를 다룰 수 있는 것.
코드를 수정하지 않고 기능을 확장할 수 있다.

부모타입의 변수에 넣어도 자식클래스에 오버라이딩된 메서드가 실행된다.




추상화
중요한 공통점만 뽑아내고, 불필요한 세부부분은 감추는 것
추상클래스
공통된 특징은 가지고 있지만, 그 자체로 완전한 객체가 되기 어려운 클래스
- 공통적인 필드
- 공통적인 메서드

자식클래스들이 추상클래스를 상속해서 구체적인 내용을 작성
